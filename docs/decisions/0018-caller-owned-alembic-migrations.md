# 0018. Caller-owned Alembic migrations for the effaced-owned tables

- **Status:** accepted
- **Date:** 2026-06-12

## Context

effaced owns four tables — `effaced_audit_events`, `effaced_consent_records`,
`effaced_outbox`, `effaced_restriction_records` — mounted onto the caller's
`MetaData` by `bind_tables()` (#10), which executes no DDL and assumes no
migration tool. What was missing (#106) is the production story: how those
tables ride a host application's *real* Alembic setup, and how owned-schema
changes reach deployed databases when an effaced release alters them. This
is a production-readiness blocker for the dogfood gate (#51).

Forces:

- An effaced-shipped migration chain would have to splice into an unknowable
  host revision graph and contend for `alembic_version` ownership — every
  host resolves that differently, so any choice we hard-code is wrong for
  half of them.
- Autogenerate noise (perpetual spurious diffs on the owned tables) would
  erode the caller's trust in their own migration tooling.
- ADR 0013 created one additive-migration precedent: `effaced_outbox.operation`
  carries a server default so the additive `ALTER TABLE` backfills populated
  outboxes. Issue #106 asks for that pattern to be generalized.
- Non-negotiable #6 (manifest format) requires old manifests to load forever;
  the relationship between `MANIFEST_SCHEMA_VERSION` bumps and DB migrations
  was undocumented.

## Decision

**The caller owns migrations.** effaced never ships Alembic migration
scripts, never depends on Alembic at runtime, and never reads or writes
`alembic_version`. `bind_tables()` on the host's `target_metadata` is the
entire integration: `alembic revision --autogenerate` discovers the owned
tables like first-party models, and the caller reviews and applies the
revision like any of their own. Hosts without a migration tool keep
`metadata.create_all()`.

**The library-side contract that makes caller ownership safe** (binding on
every future change to the owned tables):

1. Owned-table DDL evolves **additively only** in MINOR releases — new
   tables, columns, or indexes. Dropping, renaming, or retyping anything is
   MAJOR with a documented upgrade path (widened SemVer, ADR 0003).
2. A new column on an owned table must be **nullable or carry a
   `server_default`**, so the caller's autogenerated additive `ALTER`
   backfills populated tables in place — the ADR 0013 pattern, generalized
   (every owned table may hold rows in production).
3. Index names stay explicit and stable (≤63 chars), immune to caller
   `naming_convention`s.
4. **Round-trip cleanliness:** after applying, a second autogenerate yields
   zero diffs — including under `compare_type` and `compare_server_default`
   — pinned by `test_alembic_autogenerate_pg.py`. No perpetual-diff noise.
5. Every owned-table DDL change ships a changelog note instructing callers
   to re-run `alembic revision --autogenerate` after upgrading.

**Manifest-version upgrades stay orthogonal.** Manifests are serialized JSON
held by the caller, never rows in the owned tables; `migrate()` lifts old
payloads forward in memory. A `MANIFEST_SCHEMA_VERSION` bump alone therefore
never requires a DB migration (the v1→v2 precedent, `RetentionPolicy.anchor`
in ADR 0012, touched zero DDL). When one release bumps both, the caller's
recipe is unchanged — upgrade the package, autogenerate, review, apply — and
the changelog calls out both.

**Opt-out:** callers who manage `effaced_*` DDL out-of-band exclude the
tables from autogenerate with an `include_object`/`include_name` hook; the
snippet lives in the Alembic guide.

## Consequences

- effaced cannot control *when* a host applies migrations; the contract is
  that whatever autogenerate emits for the owned tables is always a safe
  additive change, and the review step stays with the caller — consistent
  with shipping mechanisms, never determinations.
- `test_bind_tables.py::test_no_server_defaults_except_the_outbox_operation_migration_aid`
  pins the *current* server-default set; a future additive column may
  legitimately extend its exception list — in the same PR, citing this ADR.
- A new integration surface must stay green: the autogenerate round-trip
  test fails the moment a column, type, or index change breaks dialect
  round-tripping. That is the point.
- Alembic enters the root dev dependency group only — never the `effaced`
  runtime dependencies.
