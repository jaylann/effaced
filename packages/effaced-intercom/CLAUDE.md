# effaced-intercom

First-party Intercom resolver. Depends on `effaced` (workspace source) + `httpx` only — no Intercom SDK (direct httpx keeps the test boundary clean, faked at the transport layer).

- `IntercomResolver` implements the `Resolver` protocol (structural; `test_intercom_conformance.py` runs the shared `effaced.testing.ResolverConformanceSuite`). Keep `name == "intercom"` stable forever — it's recorded in audits and outbox entries.
- Refs: `kind="intercom"`, `value=<Intercom internal contact id>` — get/delete address the contact by id (`/contacts/{id}`), so erasure needs no email search and no enumeration. The id is sent verbatim, percent-encoded into a single path segment (`quote(value, safe="")` — wrong-target-erasure guard, `_segment`); never normalize or log it.
- **Idempotency:** Intercom 404 on get/delete ⇒ absent — export returns empty, erasure reports `already_absent=True`; success, not an error. Transient faults (429, 5xx, connection) propagate for saga retry; other 4xx raise `ResolverError` (taxonomy table in `errors.py`, status-code-only — Intercom error bodies are versioned, the status is the stable part). `ResolverError` messages carry only status + action verb, never the id or token.
- Export is contact `email` (CONTACT), `name` (IDENTITY), `phone` (CONTACT), plus per-conversation `created_at`/`updated_at`/`state` (BEHAVIORAL — interaction logs) from `POST /conversations/search` filtered by `contact_ids`, **paged** via `pages.next.starting_after` (per-page 150). Conversation records are keyed `conversation.{id}.{field}`. The field set lives in `export_records.py` and is behaviour under widened SemVer: removing/recategorising a field is MAJOR, adding is MINOR.
- **Never exported:** message bodies (`conversation_parts`, `source.body` — deep COMMUNICATION content, deliberately untouched) and the caller-defined `custom_attributes` blob. Empty-string/absent fields mean "not held" — skipped.
- **Covered surface:** `covered_surface` (the `AttestingResolver` capability) returns `INTERCOM_COVERED_SURFACE` (`covered_surface.py`), built FROM `export_records._CONTACT_FIELDS` + `_CONVERSATION_FIELDS` — the same tuples the exporter iterates — so declaration and exporter cannot drift. Conversation ids are dynamic, so conversation fields are declared as globs (`conversation.*.created_at`); the concrete `conversation.{id}.created_at` rows match. Exclusions: `contact.custom_attributes.*`, `conversation.*.conversation_parts.*`. `make_fully_populated_resolver` seeds a contact with every profile field + ≥1 conversation so enumeration exercises every glob. Declaration of claimed reach, never a compliance determination.
- **Erasure does NOT separately delete conversations** — `DELETE /contacts/{id}` removes the contact; conversation retention is Intercom's. Documented in README.
- **No `rectify_subject`** in this first cut — consistent with resend; resolvers without it are skipped (never an error) during rectification runs. Don't add it without a field-keyed correction vocabulary.
- Sync `httpx` wrapped in `asyncio.to_thread`; a fresh client per call — nothing loop- or connection-bound cached on the instance (ADR 0006). The conversation-search pagination loop opens one client for the whole logical call (created inside the call, not cached).
- No live network calls in tests; inject `transport=FakeIntercomTransport(...)` (`tests/fake_intercom_transport.py`) — faking at the httpx transport boundary exercises the real request pipeline (URL construction, headers, JSON decode, pagination).
